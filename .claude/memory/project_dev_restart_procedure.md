Restarting `npm run dev` mid-session (e.g. after rebuilding `shared` with `npm run build --workspace=shared`, which races with `tsx watch` and can leave it in `EADDRINUSE`) repeatedly corrupted Vite's dependency pre-bundling cache (`node_modules/.vite`, `client/node_modules/.vite`), producing `Uncaught ReferenceError: exports is not defined` in the browser. This happened multiple times before it was fixed properly.

**The fix**: `npm run kill` (`scripts/killDev.sh`) kills every process the dev stack can spawn AND clears both `.vite` cache directories. `npm run doctor` (`scripts/devHealthcheck.sh`) now also detects a stacked/duplicate `npm run dev` invocation (not just down servers) and calls `killDev.sh` automatically.

Process-tree gotchas that made earlier hand-rolled `pkill`/`ps aux` attempts unreliable:
- `tsx watch` spawns a grandchild worker with a completely different command line (`node --require .../tsx/dist/preflight.cjs ...`) that a `pkill -f "tsx watch src/index.ts"` pattern does NOT match — it survives as an orphan unless killed via the `tsx/dist/preflight.cjs` pattern too.
- Counting steady-state vite/tsx worker processes to detect duplicates is racy: a crash-looping second instance (fighting over a port) may not have a settled worker process at the moment you check, so the count silently misses it.
- `pgrep -f "node_modules/.bin/concurrently"` is the robust duplicate-detection signal instead: `npm run dev` spawns exactly one `concurrently` process per invocation, and it keeps running even while its children (vite/tsx) are actively crash-looping on a port conflict — so checking `concurrently` count catches a stacked second `npm run dev` reliably where checking its children does not.

**Why:** Burned significant time debugging one-off `pkill`/`ps aux` combinations live in the session, hitting the same tsx-grandchild and crash-loop-detection traps repeatedly, before finally centralizing the logic into `killDev.sh` + a corrected `devHealthcheck.sh`.
**How to apply:** Never hand-roll `pkill`/`ps aux` cleanup again in this repo — run `npm run kill`, wait for it to report done, then `npm run dev`. If writing/editing the health-check or kill scripts further, remember the concurrently-count trick for duplicate detection, not vite/tsx worker counts. **The filesystem shared between sandbox and host is flaky about delivering change events** (per the user, this has surfaced in various surprising ways before). Concretely observed here: after editing `shared/src/constants.ts`, `shared/dist` sometimes rebuilt and `tsx watch` sometimes restarted — and sometimes neither happened, silently, with no error, leaving the running server on stale compiled code indefinitely. This wasn't tied to *how* the edit was made (verified it happening even from my own `Edit` tool and my own `npm run build --workspace=shared` one-shot rebuild) — the mount itself just doesn't reliably deliver the underlying fs events that `tsc --watch`/`tsx watch`/Vite's HMR all depend on.

**Why this matters:** No watch-based tool in this stack (`tsc --watch`, `tsx watch`, Vite HMR) can be trusted to reliably pick up a change, regardless of source. Assuming "just save and the watchers will handle it" is wrong and led me to give the user incorrect guidance once already.
**How to apply:** After editing any file under `shared/src`, use `npm run restart` (kill, then `npm run dev`) to be certain the change actually took effect. To verify staleness before assuming: compare `shared/src/constants.ts` against the compiled `shared/dist/constants.js` directly (`grep` both) rather than trusting log lines like `[tsx] change in .../index.js Restarting...`, since the absence of that log doesn't reliably mean nothing needs rebuilding, and its presence doesn't guarantee the change actually landed either.

**`npm run dev` no longer runs a continuous `tsc --watch` for `shared` at all** (as of the session where the user reported spontaneous, disruptive `ws proxy error: ECONNRESET` disconnects during normal play with nothing obviously triggering them). `dev` is now `npm run build --workspace=shared && concurrently "...server..." "...client..."` — a one-shot rebuild before the concurrently processes start, not a background watcher. `restart` simplified to just `npm run kill && npm run dev`. The hypothesis (unconfirmed at time of writing): the flaky shared filesystem could fire a phantom change event at `tsc --watch`, which cascades to `tsx watch` restarting the actual Colyseus server mid-session, dropping every open WebSocket the Vite dev proxy is holding — this reads to the player as a random disconnect requiring a manual reconnect. Removing the watcher entirely is a diagnostic experiment: if disconnects stop, that confirms the watcher was the cause; if they continue, the next place to look is a real crash/exception in the server's game loop (check for uncaught errors in server logs — `tsx watch` also auto-restarts on an actual process crash, not just file changes, which would look identical to this same symptom).
**How to apply:** Don't re-add a continuous `shared` watcher to `npm run dev` without checking whether this experiment actually resolved the disconnects. If disconnects persist after this change, treat it as a real server-side bug hunt (check for unhandled exceptions/promise rejections in `gameRoom.ts`'s tick loop or Rapier physics calls), not an environment-flakiness issue.

**The loop isn't done at `npm run dev` — the browser also needs a hard refresh.** Even with a verified-fresh server process running the correct compiled code, the user still saw old behavior once because the browser tab was left open from before the restart (stale WebSocket/room state, possibly stale HTTP cache too). The user's adopted routine is "kill, dev, reload" — `npm run kill` && `npm run dev` && a hard refresh (not just re-entering the game via the app's own exit/rejoin UI) of the browser tab. All three steps matter; skipping the hard refresh reproduces the exact "server has the new code but game still behaves like the old value" confusion.
**How to apply:** When telling the user to retest after a server-side change, always mention the hard refresh explicitly, not just the kill+dev restart.

**`killDev.sh`'s `pkill -f` patterns must never match a superstring of themselves.** It used to include `pkill -9 -f "npm run dev"` as a defensive catch-all. When I added `npm run restart` (`npm run kill && npm run build --workspace=shared && npm run dev`), that script's own `sh -c` command line contains the literal substring `"npm run dev"` at the end of the chain — so `npm run kill`, as the very first thing `restart` does, matched and SIGKILLed its own parent process tree, silently aborting `restart` right after the kill step with no error. Removed that pattern entirely; `pgrep -f "node_modules/.bin/concurrently"` (already present) is the precise, non-self-matching signal for an actual running dev session.
**Why:** `pkill -f` matches anywhere in the full command line, not just the intended process — any future addition to `killDev.sh`'s patterns needs to be checked against this failure mode, especially since new package.json scripts often *do* legitimately contain other script names as substrings (e.g. `restart` containing `dev`, `kill`, and `build`).
**How to apply:** Before adding a new `pkill -f "X"` pattern to `killDev.sh`, check whether any existing or plausible future npm script's command line could contain `X` as a substring of itself. Prefer matching a specific binary path (e.g. `node_modules/.bin/foo`) over a generic phrase like a full npm-script invocation.
